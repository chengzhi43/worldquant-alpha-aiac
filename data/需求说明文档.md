# AIAC 2.0 (AIACV2): 基于 Alpha-GPT 的持续多样性 Alpha 挖掘系统 —— 需求说明文档

**版本**：v1.0
**日期**：2026-01-24
**状态**：正式版

---

## 1. 背景与目标

### 1.1 背景
当前项目已经具备基本的 Alpha 挖掘链路（LLM 生成 -> 模拟 -> 入库）。
用户希望基于 **Alpha-GPT** (Human-in-the-Loop AI) 的理念，对本项目进行系统化改造，实现“持续不断挖掘不同区域、不同数据集、不同字段组合的多样性 Alpha”，并形成稳定、可扩展、可控的自动化流程。

### 1.2 总目标 (North Star)
构建一个**人机协作（Human-in-the-Loop）的 Alpha 挖掘工厂**，实现：
1.  **稳定产出**：每天稳定挖掘 **3–4 个合格 Alpha**（模拟成功 + 满足质量门槛）。
2.  **持续多样性**：自动探索不同区域 (Region)、数据集 (Dataset)、数据字段 (Datafield) 和算子组合，避免同质化。
3.  **全链路可视化 (Web UI)**：
    *   实时监控挖掘进度、当前挖掘区域/数据集。
    *   展示挖掘次数、成功率、模拟效果。
    *   可视化 Alpha 详细流程（Idea -> Formula -> Simulation）。
4.  **主动干预与反馈 (Feedback Loop)**：
    *   支持人工调整目标偏好、质量门槛、多样性策略。
    *   **反馈闭环**：系统自动分析失败原因（如字段缺失、模拟失败），优化后续挖掘；支持由于对每日产出的抽检反馈系统。

### 1.3 核心设计理念 (Alpha-GPT Paradigm)
参考 Alpha-GPT 2.0，系统采用 **Multi-Agent 架构**：
*   **Alpha Mining Agent**: 负责挖掘（数据探索 + 表达式生成 + 模拟）。
*   **Alpha Analysis Agent**: 负责分析（归因分析 + 效果评估 + 风险检查）。
*   **Human-in-the-Loop**: 人类研究员通过 Web UI 注入直觉与偏好，系统反馈执行结果，形成正向循环。

## 2. 术语与定义
(沿用原文档定义，新增以下内容)

*   **Alpha-GPT Protocol**: 一套人机交互协议，明确人类指令（Intent）如何转化为通过 Agent 执行的任务（Task）。
*   **Feedback Loop**: 指“挖掘结果 -> 归因分析 -> 策略调整 -> 下一轮挖掘”的闭环过程。
*   **Quality Gate**: 质量门槛，包含硬性指标（Sharpe, Turnover）和软性指标（解释性、过拟合风险）。
*   **Diversity Score**: 多样性评分，衡量新挖掘 Alpha 与历史库中 Alpha 的差异程度。

## 3. 现状与问题分析 (As-Is -> To-Be)

| 维度 | 当前现状 (As-Is) | 目标状态 (To-Be) |
| :--- | :--- | :--- |
| **交互模式** | 脚本/命令行运行，无界面 | **Web UI 可视化交互**，所见即所得 |
| **挖掘策略** | 固定 Region/Universe/Dataset | **自动探索**，动态调整 Region/Universe/Dataset |

| **可观测性** | 仅日志文件 | **仪表盘 (Dashboard)**，实时展示 KPI (产出率、多样性) |

## 4. 需求概述 (Functional Requirements)

### 4.1 核心功能模块

#### 4.1.1 自动化挖掘引擎 (Auto-Mining Engine)
*   **多维探索**: 自动遍历 Region (USA, CHN, ASI, EUR...) 和 Dataset。
*   **预算管理**: 每日设定 Token 预算和模拟次数预算。
*   **智能生成**: 基于 Alpha-GPT Prompt 模板，生成符合 BRAIN 语法的表达式。
*   **规则守门**:
    *   语法校验 (Syntax Check)
    *   字段存在性校验 (Field Check)
    *   算子黑名单/白名单 (Operator Check)

#### 4.1.2 反馈闭环系统 (Feedback Loop System)
*   **失败归因 (Failure Attribution)**:
    *   识别错误类型：语法错误、模拟超时、无 Alpha ID、Metrics 不达标。
    *   将错误信息结构化，用于修正下一轮 Prompt。
*   **人工反馈 (Human Feedback)**:
    *   用户可在 UI 上对 Alpha 进行 "Like" (收藏) 或 "Dislike" (淘汰)。
    *   Dislike 原因（如“逻辑不可解释”）将被记录，用于优化 Agent。

#### 4.1.3 Web UI 可视化平台
*   **Dashboard**:
    *   今日产出概览 (Total/Valid/Selected).
    *   当前任务状态 (Mining: Region X, Dataset Y).
    *   资源消耗 (API Calls, Time).
*   **Alpha 详情页**:
    *   展示 Alpha 表达式、解释、回测指标 (Sharpe, Returns, Turnover).
    *   展示 PnL 曲线图 (需调用模拟结果接口).
*   **配置中心 (Settings)**:
    *   设置目标 Regions, Universe, Datasets.
    *   设置 Quality Thresholds (Sharpe > 1.5, Turnover < 70%).
    *   设置 Diversity Constraints.

### 4.2 业务流程

1.  **计划阶段 (Planning)**:
    *   系统根据历史数据和多样性策略，生成今日挖掘计划 (Task List)。
    *   用户可人工调整计划 (Human Intervention).
2.  **执行阶段 (Execution)**:
    *   Agent 逐个执行 Task。
    *   Step 1: 获取数据元信息 (Datafields)。
    *   Step 2: 生成 Alpha 假设与表达式 (LLM)。
    *   Step 3: 静态代码检查与修正。
    *   Step 4: 提交 BRAIN 模拟。
    *   Step 5: 获取模拟结果与日志。
3.  **评估阶段 (Evaluation)**:
    *   **Quality Filter**: 过滤掉指标不达标的 Alpha。
    *   **Diversity Filter**: 与库中 Alpha 对比，剔除高相关性 Alpha。
    *   **Result Persistence**: 入库 (Alpha Base)。
4.  **反馈阶段 (Feedback)**:
    *   更新 dataset 统计信息 (如该 dataset 是否容易出 Alpha)。
    *   更新 Prompt 策略 (如某类算子经常报错，降低使用权重)。

## 5. 详细功能需求

### 5.1 数据源与元数据管理
*   **自动同步**: 每日同步 BRAIN 平台的 Datasets 和 Datafields 元数据。
*   **Smart Selection**: 根据 Field Count, Coverage, Alpha Count 智能推荐挖掘潜力大的 Dataset。

### 5.2 Alpha 生成与优化
*   **Prompt Management**:
    *   核心 Prompt: Alpha-GPT 标准化 Prompt。
    *   Few-Shot Examples: 动态插入高质量 Alpha 示例。
    *   User Instructions: 插入用户当前的干预指令。
*   **Self-Correction**:
    *   若模拟报错，将错误信息喂回 LLM 进行自我修正 (Self-Correction Loop).

### 5.3 模拟与回测
*   **并发控制**: 控制模拟并发数，避免 429 Rate Limit。
*   **稳健性**: 处理网络波动和 API 超时。

### 5.4 结果分析与展示
*   **指标计算**: 除了 BRAIN 返回的指标，计算自定义复合打分 (Composite Score)。
*   **可视化**:
    *   挖掘漏斗图 (Generated -> Valid Syntax -> Simulated -> Good Perf -> High Diversity).

---

## 6. 非功能需求

1.  **可扩展性**: 支持接入新的 LLM 模型 (Claude, GPT-4, DeepSeek)。
2.  **数据安全**: API Key 加密存储，本地数据定期备份。
3.  **响应速度**: Web UI 操作响应 < 200ms，模拟结果异步加载。
4.  **工程规范**: 遵循 Modular Monolith 架构，前后端分离 (Python FastAPI + Modern Frontend)。
